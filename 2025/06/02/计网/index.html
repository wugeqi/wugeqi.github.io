一、概览

1.osi七层模型

应用层：负责给应用程序提供统一的接口

表示层：负责把数据转换成兼容另一个系统能够识别的格式

会话层：负责建立、管理和终止表示层实体之间的通信会话

传输层：负责端到端的数据传输

网络层：负责数据的路由、转发、分片

数据链路层：负责数据的封帧和差错检测以及MAC寻址

物理层：负责在物理网络中传输数据帧

2.tcp/ip模型

应用层：支持 HTTP、SMTP 等最终用户进程（HTTP\HTTPS\CDN\DNS\FTP)

传输层：处理主机到主机的通信（TCP、UDP）

网络层：寻址和路由数据包（IP 协议）

网络接口层：通过网络的物理电线、电缆或无线信道移动比特

二、TCP/UDP

TCP：

组成：头部+数据

头部：20/40字节

源端口号，目标端口号，序列号，确认应答号，syn/ack/fin ，校验和，选项，首部长度，窗口大小

用途：HTTP

使用

1.建立连接：三次握手

客户端主动建立连接

第一次握手：客户端发送syn=1和随机生成的初始化序列号请求建立连接，随后转成SYN_SENT状态

第二次握手：服务端在接受到请求连接的SYN报文后，其次把 TCP 首部的「确认应答号」字段填入 client_isn + 1,接着把 SYN 和 ACK 标志位置为 1,发送该报文，之后处于SYN-RCVD状态

第三次握手：客户端收到服务端报文之后还需要发送应答报文，首先该应答报文 TCP 首部 ACK 标志位置为 1 ，其次「确认应答号」字段填入 server_isn + 1，最后把报文发送给服务端

问题：

1.为什么不能两次握手？

防止历史连接，浪费资源

同步双方初始序列号：接收方可以去除重复的数据，可以按序接收

避免资源浪费：不收到ack，会重复发送syn，建立多个冗余连接

2.为什么不能四次握手？

三次握手已经很好地保证双方都有接收和发送消息的能力，采用四次握手浪费资源

3.不同阶段丢失会发生什么？

（1）重发1

（2）重发1

（3）重发2

2.断开连接：四次挥手

客户端主动断开连接

第一次挥手：客户端发送fin报文表示断开连接，进入 FIN_WAIT_1 状态

第二次挥手：服务端接受到fin报文，发送ack，服务端进入 CLOSE_WAIT 状态

第三次挥手：在等待所有未发送的消息发送完毕后，服务端发送fin报文，之后处于 LAST_ACK 状态

第四次挥手：接受到fin报文，发送ack，此时客户端将进入 TIME_WAIT 状态

服务端收到 ACK 确认包后，就进入了最后的 CLOSE 状态

客户端经过 2MSL 时间之后，也进入 CLOSE 状态

问题：

1.可以三次握手吗？

可以，在一些情况下可以把第二次挥手和第三次挥手合并起来

3.为什么需要wait状态？

这个状态的持续时间是2msl，msl指报文在网络中最多能够存活的时间，也就是发送一次报文的时间为1msl，如果服务端迟迟没有收到ack的消息的话，可以有时间进行重发，从而确保正常的关闭，所以wait状态确保信息有重发的机会

3.重传机制：

超时重传：设定一个定时器，当超过指定的时间后，没有收到对方的 ACK 确认应答报文，就会重发该数据

快速重传：当收到三个相同的 ACK 报文时，会在定时器过期之前，重传丢失的报文段。

SACK：可以将已收到的数据的信息发送给「发送方」，这样发送方就可以知道哪些数据收到了，哪些数据没收到，知道了这些信息，就可以只重传丢失的数据。

D-SACK:使用了 SACK 来告诉「发送方」有哪些数据被重复接收了。

4.滑动窗口：

5.流量控制：避免发送方的数据填满接收方的缓存

6.拥塞控制：避免发送方的数据填满整个网络

  • 慢启动：当发送方每收到一个 ACK，拥塞窗口 cwnd 的大小就会加 1。
  • 拥塞避免：当拥塞窗口 cwnd 「超过」慢启动门限 ssthresh 就会进入拥塞避免算法。每当收到一个 ACK 时，cwnd 增加 1/cwnd。
  • 拥塞发生：
    • 超时重传：
      • ssthresh 设为 cwnd/2，
      • cwnd 重置为 1
    • 快速重传
      • cwnd = cwnd/2 ，也就是设置为原来的一半;
      • ssthresh = cwnd;
  • 快速恢复
    • 拥塞窗口 cwnd = ssthresh + 3 （ 3 的意思是确认有 3 个数据包被收到了）；
    • 重传丢失的数据包；
    • 如果再收到重复的 ACK，那么 cwnd 增加 1；
    • 如果收到新数据的 ACK 后，把 cwnd 设置为第一步中的 ssthresh 的值，原因是该 ACK 确认了新的数据，说明从 duplicated ACK 时的数据都已收到，该恢复过程已经结束，可以回到恢复之前的状态了，也即再次进入拥塞避免状态；

ipv6：

128位，每16位作为一组，每组用：隔开

优点：ipv6可自动配置，ipv6包头首部长度采用固定的值40字节，去掉了包头校验和，提高了传输的性能，ipv6有应对伪造ip地址的网络安全功能以及线路窃听的功能提高了安全性。取消了分片/重新组装相关字段，只在源与目标主机进行分片与重组

类型：

组播：一对多

协议：

DNS协议：

流程：

1.浏览器首先看下自己的缓存里有没有，如果没有就看看操作系统里的缓存有没有，其次检查本机域名解析文件有没有，最后通过DNS服务器进行查询

2.客户端发送DNS请求，问本地DNS服务器

3.本地DNS服务器有直接返回，如果没有则向根DNS服务器请求

4.根DNS服务器收到请求后，发送对应的顶级域名服务器给本地DNS服务器

5.本地DNS服务器接收到后，再向对应的顶级域名服务器发送请求

6.顶级域名DNS服务器收到请求后，发送对应的权威服务器给本地DNS服务器

7.本地DNS服务器接收到后，再向对应的权威服务器发送请求

8.本地 DNS 再将 IP 地址返回客户端，客户端和目标建立连接

ARP协议：得到对应的MAC地址

流程：

主机会通过广播发送 ARP 请求，这个包中包含了想要知道的 MAC 地址的主机 IP 地址。

当同个链路中的所有设备收到 ARP 请求时，会去拆开 ARP 请求包里的内容，如果 ARP 请求包中的目标 IP 地址与自己的 IP 地址一致，那么这个设备就将自己的 MAC 地址塞入 ARP 响应包返回给主机。

DHCP:动态获取ip地址

流程：

客户端首先发起DHCP发现报文的ip数据报，由于客户端没有ip地址，也不知道DHCP服务器的地址，所以使用的是UDP广播通信。DHCP 客户端将该 IP 数据报传递给链路层，链路层然后将帧广播到所有的网络中设备。

DHCP 服务器收到 DHCP 发现报文时，用 DHCP 提供报文（DHCP OFFER） 向客户端做出响应。该报文仍然使用 IP 广播地址 255.255.255.255，该报文信息携带服务器提供可租约的 IP 地址、子网掩码、默认网关、DNS 服务器以及 IP 地址租用期。

客户端收到一个或多个服务器的 DHCP 提供报文后，从中选择一个服务器，并向选中的服务器发送 DHCP 请求报文（DHCP REQUEST进行响应，回显配置的参数。

服务端用 DHCP ACK 报文对 DHCP 请求报文进行响应，应答所要求的参数。

如果租约的 DHCP IP 地址快期后，客户端会向服务器发送 DHCP 请求报文：

服务器如果同意继续租用，则用 DHCP ACK 报文进行应答，客户端就会延长租期。

服务器如果不同意继续租用，则用 DHCP NACK 报文，客户端就要停止使用租约的 IP 地址。

NAT：将私有地址转换成公共地址

ICMP:互联网控制报文协议，确认 IP 包是否成功送达目标地址、报告发送过程中 IP 包被废弃的原因和改善网络设置等

1.为什么进行子网的划分？

如果两台计算机要通讯，首先判断是否在同一个广播域中，即网络地址是否相同。如果网络地址相同，表明接受方在本网络上，那么可以把数据包直接发送到目标主机。路由器寻址工作中，也就是通过这样的方式来找到对应的网络号的，进而把数据包转发给对应的网络内。

四、HTTP：超文本传输协议

组成：

请求报文：

请求行：请求方法、请求目标（url)和协议版本

请求头部：包含请求的附加信息，如host，user-Agent、content-Type

请求体：包含请求的数据

响应报文：

状态行：包含HTTP协议版本、状态码和状态信息

响应头部：包含关于响应的附加信息，如Content-Type、Content-Length

响应体：包含响应的数据

状态码:

1xx:提示信息，表示目前是协议处理的中间状态，还需要后序的操作

2xx:指成功执行 200

3xx:指资源位置发送变动，需要重新请求 301（指永久重定向）302（指临时重定向）

4xx:指客户端发送的报文出现问题，服务端无法处理 404（资源不存在，url错了）405（请求的方法类型不支持）

5xx:指服务端处理时内部出现问题 502（服务器作为网关/代理，从上游服务器接收到无效的响应） 504（服务器作为网关/代理，未能及时从上游服务器收到响应

请求类型：

GET：用于获取指定资源，通常用于获取数据

POST：用于向服务器提交数据

发展：

HTTP和HTTPS

区别：

HTTP是超文本传输协议，信息是明文传输，存在安全风险的问题，HTTPS使用了SSL/TLS协议，变得更加安全，信息加密、校验机制、身份证书

HTTP连接建立相对简单，TCP三次握手后即可以进行HTTP的报文传输，而HTTPS还需要SSL/TLS的握手过程

默认的端口不一致：HTTP:80,HTTPS:443

HTTPS协议还需要向CA申请数字证书，确保服务器安全可靠

SSL/TLS握手过程？

客户端发送

第一次握手：发送加密通信请求，主要发送一个随机数，支持的TLS协议版本，支持的密码套件

第二次握手：收到请求后，向客户端发出响应，校验客户端发来的TLS协议版本，判断是否能够支持，如果不能支持关闭加密通信，如果能够支持，发送一个随机数，确认的密码套件和数字证书

第三次握手：客户端收到服务端的回应之后，校验服务端发来的数字证书是否正确，如果正确的话，生成一个随机数，加密通信算法改变通知，之后进行加密传输，客户端握手结束通知，表示客户端的握手阶段已经结束，并把之前发送的内容的数据做个摘要供服务端检查

第四次握手：服务器在接收到所有的信息后，加密通信算法改变通知，之后进行加密传输，服务端握手结束通知，表示服务端的握手阶段已经结束，并把之前发送的内容的数据做个摘要供客户端检查

如何有这些优势的？

信息加密：https 握手期间会通过非对称加密的方式来协商出对称加密密钥，根据加密密钥进行通信

校验机制：无法篡改通信内容，篡改了就不能正常显示

身份证书：具有公证力的数字证书供校验

HTTP比HTTPS快吗？

HTTP1.0

HTTP1.1:

HTTP采用请求-应答模式，客户端发起请求，然后服务端做出响应，且HTTP是基于TCP传输协议实现的，在进行HTTP通信前，要建立TCP连接，然后发送HTTP请求，服务端返回响应之后释放TCP连接，如果在每次发送HTTP请求时都需要建立连接，这就是短连接

长连接指可以使用同一个TCP连接来发送和接收多个HTTP请求/应答

HTTP2.0：1.头部压缩2.二进制格式3.并发传输，引用了Stream概念。多个Stream复用一个TCP连接，解决了队头堵塞问题4.服务器主动推送资源

HTTP3.0：1.连接快2.连接迁移3.无队头堵塞
